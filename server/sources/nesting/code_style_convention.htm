Extreme Nest code style convention

  1. Language standard

    We use the latest Visual Studio 2017 version for development and a C++ compiler which supports C++ 14.

  2. File names

    1. Source code file names and extensions

      Source code should be stored in files either with .cpp or .h extensions. File names should use snake_case naming convention.

  3. Formatting

    1. Spaces should be used in source and header files. Tabs symbols are prohibited.
    2. Tab size should be equal to 4 space symbols.
    3. Local scopes ({} statements) should be located each at new line.
    4. Don't use space symbols at the end of lines.
    5. Empty lines should not contain space symbols.
    6. Each source or header file should contain one empty line at the end of file.
    7. Instead of using macros, #pragma once directive should be used.
    8. Usually comments inside source code should begin with double slash (//) symbols, then a space should come. Text should begin with a capital letter, like that: // Compress points. Comments could be at the end of the source code line and should be separated by a space from the source code: 
      int i = 1; // Comment text
    9. Use unsigned keyword instead of unsigned int.
    10. For pointer and reference types use type_name* var_name or type_name& var_name scheme.

  4. Conventions of file inclusions

    The file inclusions should be organized in fhe following order:

    1. Include system headers, like 
      #include <vector>
    2. Include widely used libraries, like Boost: 
      #include <boost/mpl/list.hpp>
    3. Include Extreme Nest specific libraries, like 
      #include "protocol.h"
    4. Include all other local headers.

  5. Naming convention

    1. Namespaces

      Namespace names should be in PascalCase naming convention. If namespace name is contraction then all letters should be in upper case. Prefer names in singular form rather than plural names. For instance, use Network::Container instead of Network::Containers.

    2. Interfaces

      Interface names should be in PascalCase naming convention starting with capital I letter. For instance, IMemoryBuffer.

    3. Interface methods

      Interface methods should use camelCase naming convention.

    4. Classes and structures

      Classes and structures should use PascalCase naming convention.

    5. Class and structure methods

      Class and structure methods should have the same names as corresponding interface methods. However, if a method is not a part of any interface, then camelCase should be used.

    6. Arguments

      Arguments for methods, interface methods, functions, constructors and destructors should use snake_case naming convention.

    7. Local variables

      Local variables should use snake_case naming convention.

    8. Class and structure fields

      Class and structure fields should use snake_case naming convention starting without any prefix. For instance, buffer. If field name conflicts with argument name then add "_" suffix to the argument name, for instance, buffer_.

    9. Template parameters

      Template parameters should use PascalCase naming convention. For instance, Vector.

    10. Standalone functions

      Standalone functions should use camelCase naming convention.

    11. Enumerators

      Enumerators should be declared inside structures, classes or namespaces. They should use PascalCase naming convention without any prefix. Enumerator members also should use PascalCase naming convention without any prefix.

    12. Constants

      13. Constants could be declared inside structures, classes or namespaces. They should use MACROS_CASE naming convention.

    13. Macros

      Macros should be used where they are really needed. They should use MACROS_CASE naming convention.

    14. Typedefs

      Typedefs should use snake_case naming convention.

  6. Coding recommendations

    1. Const specifier

      The C++ "const" specifier should be used where it is possible.

    2. Override and final specifiers

      The C++ "override" and "final" specifiers should be used where it is possible.

    3. C-style type casting

      The C-style type casting is prohibited, instead use C++ style type casting, for instance, static_cast<> or other.

  7. Exception safety

    Code should be exception safety as much as possible. We should satisfy at least basic guarantee.

    1. New and delete statements

      New and delete statements are prohibited, except case when new statement is a part of smart pointer initialization expression. Instead consider to use std::vector for temporary buffers or other smart buffer types. Consider to use factories which return smart pointers. Use smart pointers instead of raw pointers. Use raw pointers only when you need to have an non-owning pointers which are not required to be deallocated. Use boost::scoped_ptr with scope lifetime objects.

    2. FILE* pointers

      FILE* pointers are prohibited. Instead consider to use std::fstream to work with files.

  8. Thread safety

    We should allow running our code in parallel environment where it is possible.

    1. Global variables

      Don't use global variables where it is possible. Always consider to use boost::thread_specific_ptr wrapper. Use g_ prefix for global variables.

  9. Glossary

    1. PascalCase naming convention

      Naming convention where names start with a capital letter. Name can consist from several words and each word starts with capital letter. Other letters are small. Some examples are MemoryBuffer, TriangleFan, CreateMemoryBuffer.

    2. camelCase naming convention

      Naming convention where names start with a small letter. Name can consist from several words and each word starts with capital letter (except first letter). Other letters are small. Some examples are memoryBuffer, triangleFan, createMemoryBuffer.

    3. snake_case naming convention

      Naming convention where all letters in a name are small. Name can consist from several words and each word is separated from other words by using under-scope symbol. Some examples are memory_buffer, triangle_fan, create_memory_buffer.

    4. kebab-case naming convention

      Naming convention where all letters in a name are small. Name can consist from several words and each word is separated from other words by using hyphen symbol. Some examples are memory-buffer, triangle-fan, create-memory-buffer.

    5. MACROS_CASE naming convention

      Naming convention where all letters in a name are capital. Name can consist from several words and each word is separated from other words by using under-scope symbol. Some examples are MEMORY_BUFFER, TRIANGLE_FAN, CREATE_MEMORY_BUFFER.
